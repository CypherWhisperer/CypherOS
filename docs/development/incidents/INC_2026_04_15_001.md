## Summary

Running `nixos-rebuild switch` on `nixos-unstable` triggered source builds of `terraform`, `vault`, and `n8n`. On an 8GB machine with GNOME consuming ~2GB baseline, parallel multi-core builds exhausted available memory. The kernel OOM killer fired and terminated the terminal mid-build. The machine was left in an inconsistent state — the rebuild had neither succeeded nor cleanly failed. This made it impossible to safely test subsequent changes.

## Root Cause Analysis

Three root causes combined to produce the OOM condition:

**1. `nixos-unstable` binary cache lag**

When running `nixos-rebuild switch`, Nix checks `cache.nixos.org` for a pre-built binary (a _substitute_). If found, it downloads it. If not — which happens frequently on `nixos-unstable` because the cache lags the channel by hours to days — Nix **builds from source locally** without warning.

`terraform`, `vault`, and `n8n` are the worst offenders:

- They are Go/Node.js projects that compile or bundle everything from scratch
- Go compilation is memory-hungry — it links large binaries with significant working-set memory
- `n8n` in particular pulls a massive Node.js dependency tree through its build process

**2. Default Nix parallelism**

Nix defaults to `max-jobs = auto`, which means one build job per CPU core. On the 7th-gen i7 (4 cores / 8 threads), that is up to 8 parallel builds. On an 8GB machine where GNOME already consumes ~2GB, 8 parallel Go builds is a death sentence for available memory.

**3. No effective swap**

The `@swap` BTRFS subvolume was mounted at `/swap` and a 10GB swapfile existed on disk. However, `swapDevices` in `hardware-configuration.nix` was empty — NixOS had never been told about the swapfile and was not activating it at boot. The machine was effectively running with zero swap.

Without swap, when memory fills, the kernel has no relief valve. The OOM killer fires immediately rather than writing pages to disk and continuing.

**The failure mode:**

```
nixos-rebuild switch
  → cache miss on terraform, vault, n8n
  → 3 source builds queued
  → Nix spawns 8 parallel jobs (auto)
  → RAM fills: GNOME (2GB) + parallel Go builds (~6GB)
  → No swap to drain to
  → OOM killer fires, kills the terminal
  → Rebuild state: unknown
```

## Resolution

Two solution axes applied together — a belt-and-suspenders approach.

### Axis 1 — Resource Ceiling

Throttle how aggressively Nix uses hardware during builds, leaving headroom for the active DE session:

```nix
nix.settings = {
  # max-jobs: max derivations building in parallel.
  # Default "auto" = one per CPU core = 8 on the i7-7th gen.
  # Setting to 2 means at most 2 derivations build simultaneously.
  max-jobs = 2;

  # cores: how many CPU cores each build job may use.
  # Nix passes this as $NIX_BUILD_CORES to the builder.
  # 2 jobs × 2 cores = 4 cores max, leaving headroom for the DE.
  cores = 2;
};

# Runs the Nix build daemon at idle CPU and I/O scheduling priority.
# Builds proceed in the background without competing with the active session.
nix.daemonCPUSchedPolicy = "idle";
nix.daemonIOSchedClass   = "idle";
```

|Setting|Value|Effect|
|---|---|---|
|`max-jobs`|2|Max 2 derivations building in parallel (was auto = 8)|
|`cores`|2|Each job gets 2 cores max. Worst case: 4 cores total|
|`daemonCPUSchedPolicy`|idle|Build daemon only runs when nothing else needs CPU|
|`daemonIOSchedClass`|idle|Build daemon yields disk I/O to the active session|

> **Note for future hardware:** On a machine with 16GB+ RAM, these can be relaxed — `max-jobs = 4`, `cores = 4` is reasonable. The values here are tuned for 8GB RAM + active DE session.

### Axis 2 — Substitute-or-Fail

Instead of silently falling into a RAM-crushing source build on a cache miss, Nix now hard-errors and gives you a choice:

```nix
nix.settings = {
  substituters = [
    "https://cache.nixos.org"
  ];
  trusted-public-keys = [
    "cache.nixos.org-1:6NCHdD59X431o0gWypbMrAURkbJ16ZPMQFGspcDShjY="
  ];

  # Do NOT silently fall back to a source build on cache miss.
  # Error instead — forces an explicit decision.
  fallback = false;
};
```

With `fallback = false`, a cache miss produces a clear error:

```
error: a 'x86_64-linux' with features {} is required to build
'...terraform...', but I am not a builder for that platform
```

This tells you _exactly_ which package has no substitute, and you make an informed decision: wait for the cache to catch up, or explicitly allow the build with `--option fallback true`.

### Swap — Activated as a Direct Consequence

The absence of effective swap was addressed simultaneously. See [ADR-003](https://claude.ai/project/decisions/ADR-003-swap-activation.md) and [ADR-004](https://claude.ai/project/decisions/ADR-004-zram-setup.md) for the full decisions. The net result: ZRAM at 50% RAM as the primary swap device (compressed, in-memory, high priority), and the `@swap` swapfile activated as the disk-backed last resort.

## Practical Workflow After This Change

```bash
# Normal rebuild — safe. Errors instead of crashing on cache miss.
sudo nixos-rebuild switch --flake .#cypher-nixos

# When a cache miss error appears and you're ready to let it build
# (e.g. late at night, machine idle, patience available):
sudo nixos-rebuild switch --flake .#cypher-nixos --option fallback true

# Check if a package has a cached substitute before committing to a rebuild:
nix path-info --store https://cache.nixos.org nixpkgs#terraform 2>/dev/null \
  && echo "CACHED" || echo "NOT CACHED — will build from source"
```

## Lessons

- **`nixos-unstable` + source builds + constrained RAM + no swap = OOM.** Any one of these is acceptable. All four together is a crash.
- **`swapDevices = [ ]` in `hardware-configuration.nix` does not mean "no swap device."** It means NixOS was never told about the one that exists. Always verify swap is actually active after install: `swapon --show`.
- **`fallback = false` is the "tell me, don't surprise me" setting.** A controlled error is far better than a silent RAM-exhausting build.
- **`daemonCPUSchedPolicy = "idle"` and `daemonIOSchedClass = "idle"` make builds genuinely background.** The DE stays responsive even when Nix is doing work.
